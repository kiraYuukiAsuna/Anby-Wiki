# ADR-0013：延后 Beta 门禁但继续 P1 研发

状态：已接受
日期：2026-07-23

## 背景

M7-T09 的自动化、容量、恢复、OCI 和真实运行态门禁已经通过，但 Next.js
`16.2.11` 仍依赖存在 production high 漏洞的 `sharp ^0.34.5`。M7-T10 还缺正式
域名、OIDC 凭据、Beta 范围和观察期。实施方案原本要求 P0 Beta 稳定后才进入 M8。

继续等待上游会阻塞不依赖生产发布的 P1 设计、领域实现和测试。项目负责人明确决定
暂时跳过该进入条件并继续后续方案迭代。

## 决策

- M7-T09 和 M7-T10 保持未完成，不把已知 blocker 改写为 PASS。
- 允许开始 M8/M9 的设计、代码、迁移和自动化测试。
- 该例外只改变研发排序，不是安全风险豁免，也不授权生产或 Beta 发布。
- M8/M9 必须保持 AST v1、Revision 不可变、领域服务权威写入和生成客户端等 P0
  不变量；需要改变公共契约时仍按高冲突资源规则单独申请和评审。
- 正式 Beta、M8/M9 发布或对外部署前，必须重新通过 M7-T09，并完成 M7-T10 的
  真实身份接入和观察期。

## 风险与补偿控制

- P1 代码可能在真实 Beta 反馈前形成返工；以聚焦变更和 Adapter 边界降低耦合。
- `sharp` 漏洞继续作为发布 blocker；不使用 `npm audit fix --force`，不签发无期限
  漏洞例外。
- OIDC 缺失时只允许测试身份和自动化环境，production 配置门禁保持不变。
- P1 功能默认不进入 production Compose 的启用路径，直至 M7 门禁恢复为绿。

## 退出条件

- Next.js 官方依赖兼容已修复的 `sharp`，或另有经过评审的有期限风险处置；
- 正式域名和 OIDC 配置可用；
- M7-T09 全部门禁通过，M7-T10 完成约定观察期。

## 影响

- `Docs/CurrentImplementationStatus.md` 必须区分研发完成与生产发布就绪，
  不用单一“里程碑完成”状态掩盖两者差异。
- 发布阻塞统一维护在 `Docs/OutstandingIssues.md`。
